# 「热读」AI 内容管线技术方案

> 配套文档：PRD.md（v1.1，§2 核心壁垒）｜ MVP.md（v1.0，§2.4 内容引擎）
> 版本 v1.0 ｜ 状态：待评审 ｜ 对应里程碑：M1 内容引擎 MVP（关键路径）
> 目标：把"采集→去重→分类→聚类→摘要→评分→质检→发布"这条链路细化到**可开发、可验收**的程度，攻下内容质量这条生死线。

---

## 0. 设计目标与约束

### 0.1 三条铁律（不可违背）
1. **不杜撰**：任何数字、引语、事实必须来自原文；缺失则留空，绝不编造。
2. **可回溯**：每条事件必须挂得到原文链接与来源等级，事实与 AI 解读分区展示。
3. **可降级**：任一 AI 环节失败，链路可回退到"规则兜底 / 进人工待审"，绝不让脏内容直接发布。

### 0.2 质量目标（M1 出口标准）
| 指标 | 目标 | 测量方式 |
|---|---|---|
| 分类准确率 | ≥ 90% | 每日抽检 50 条人工核对 |
| 去重/合并准确率 | 重复率 < 5%，串卡率 < 3% | 抽检聚类簇 |
| 摘要事实准确率 | ≥ 95%（无幻觉、无错数字） | 人工抽检 + 引用校验通过率 |
| 端到端时延 | 单事件 采集→可发布 < 15 min | 管线埋点 |
| 日产能 | 四模块合计 ≥ 80 条候选 / 日 | 发布量统计 |

## 1. 总体架构

```
                    ┌─────────────────────────────────────────────┐
                    │              调度层 (Celery/Airflow)          │
                    └─────────────────────────────────────────────┘
                                        │ 编排
  ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
  │1 采集   │→ │2 清洗   │→ │3 分类   │→ │4 聚类   │→ │5 摘要   │→ │6 评分   │
  │ 多源    │   │ 去重    │   │ 四模块  │   │ 事件簇  │   │ 解读    │   │ 排序    │
  └────────┘   └────────┘   └────────┘   └────────┘   └────────┘   └────────┘
       │            │            │            │            │            │
       ▼            ▼            ▼            ▼            ▼            ▼
  ┌─────────────────────────────────────────────────────────────────────┐
  │   PostgreSQL（业务+元数据） + pgvector（向量） + Redis（榜单/缓存）     │
  └─────────────────────────────────────────────────────────────────────┘
                                        │
```

每个阶段产出落库并打状态标记（`raw → cleaned → classified → clustered → summarized → scored → reviewing → published / rejected`），便于断点重跑与排障。

## 2. 阶段一：信源采集

### 2.1 信源分级（沿用 PRD §2.2）
| 等级 | 类型 | 权重 | 用途 |
|---|---|---|---|
| **S** | 官方公告、监管机构、交易所、公司财报、头部财经媒体 | 1.0 | 事实主源，可单源成卡 |
| **A** | 垂直专业媒体、券商研报、知名分析师 | 0.7 | 解读与交叉验证 |
| **B** | 自媒体、社交平台（X/微博/即刻）热议 | 0.3 | **仅作热度信号**，须 S/A 交叉才可成卡 |

### 2.2 各模块种子信源（MVP 起步，每模块 ≥ 10）
| 模块 | S 级示例 | A 级示例 | B 级（热度） |
|---|---|---|---|
| 科技 | 公司官方、网信办/工信部、交易所公告 | 36Kr、晚点、The Information | 科技类 KOL、即刻 |
| 金融 | 证监会、央行、交易所、上市公司公告 | 财联社、彭博、券商研报 | 雪球热帖 |
| AI | 厂商官方博客、arXiv、GitHub Trending | 机器之心、量子位 | AI 社区/X |
| 宏观 | 统计局、央行、财政部、新华社 | 第一财经、路透 | 财经大 V |

> 信源清单维护在配置表 `sources`，可后台增删改、调权重、停启用。

### 2.3 采集方式
| 方式 | 适用 | 技术 |
|---|---|---|
| RSS | 有 RSS 的媒体/博客 | `feedparser` 定时拉取 |
| 媒体/新闻聚合 API | 财经/科技新闻 | 第三方 API（按额度） |
| 定向爬虫 | 公告、研报、官方站 | Scrapy + Playwright（动态页） |
| 社交信号 | 热度榜、话题 | 平台 API / 受控抓取（仅取热度，不全文） |

### 2.4 采集要点
- **频率**：S 级 5–15 min 轮询，A/B 级 30–60 min；突发事件可手动触发。
- **去重前置**：以 `url + 标题指纹` 做入库幂等，避免重复抓取。
- **合规**：遵守 robots、限流、仅抓摘要+回源；B 级只取热度不存全文。
- **原始落库**：`raw_articles`（url, source_id, title, content, published_at, fetched_at, raw_hash）。

## 3. 阶段二：清洗与去重

### 3.1 清洗
- **正文抽取**：`trafilatura` / `readability` 去导航、广告、模板。
- **规范化**：去 emoji 噪声、统一全半角、抽取发布时间、提取数字与实体（公司/人物/金额）。
- **质量门槛**：正文 < 80 字、纯营销稿、404 → 丢弃或降级。

### 3.2 去重（两级）
| 级别 | 方法 | 阈值 | 作用 |
|---|---|---|---|
| **精确去重** | 标题/正文 SimHash 或 `raw_hash` | 完全一致 | 拦截转载、镜像 |
| **语义去重** | Embedding 余弦相似度（pgvector） | cos ≥ **0.92** 视为同文 | 拦截改写、洗稿 |

> Embedding 模型：用一款中文友好的句向量模型（如 bge / m3e 系列），维度入 pgvector，建 IVFFlat/HNSW 索引。
> 去重保留策略：同组保留**来源等级最高 + 最早发布**者为主文，其余作为该事件的附加来源。

## 4. 阶段三：分类（四模块 + 标签）

### 4.1 方案：LLM 主分类 + 规则兜底
- **规则兜底（先跑）**：关键词/信源映射（如来自证监会→金融）先给一个候选，降低 LLM 调用量与错误。
- **LLM 分类**：对规则不确定的，调用 LLM 输出 `{module, tags[], confidence}`。

### 4.2 分类 Prompt（模板）
```
你是「热读」的内容分类器。请把下面这篇资讯归入唯一一个主模块，并给出细分标签。

主模块（四选一）：科技 / 金融 / AI / 宏观
细分标签从对应模块标签库中选 1-3 个，没有合适的可返回空。

【标签库】
科技：硬件/出海/平台/政策/消费电子
金融：A股/美股/一级市场/加密/银行
AI：大模型/Agent/芯片算力/应用层/论文
宏观：货币/财政/地产/出口/就业

【输出 JSON】
{"module":"", "tags":[], "confidence":0-1, "reason":"一句话理由"}

【资讯】
标题：{title}
正文：{content_truncated}
```

### 4.3 兜底规则
- `confidence < 0.6` → 进**人工待审**队列，不自动发布。
- 跨模块强相关（如"AI 芯片融资"）→ 主模块取最相关，其余作为标签关联。

## 5. 阶段四：事件聚类

### 5.1 目标
把"同一件事的 N 篇报道"聚成**一个事件卡片**（多源），而非 N 张重复卡。

### 5.2 方法（增量聚类，适合流式）
1. 新文章生成 Embedding。
2. 在**最近 72 小时**同模块事件簇质心中检索最相似簇。
3. 若 `cos ≥ 0.86` → 并入该事件簇（更新质心、追加来源）；否则**新建事件簇**。
4. 离线每日用 HDBSCAN 复核，纠正错聚（拆/并）。

| 参数 | 取值 | 说明 |
|---|---|---|
| 时间窗 | 72h | 超窗的同主题视为新事件（如"每月 CPI"） |
| 并簇阈值 | cos ≥ 0.86 | 偏保守，宁可多簇也别串卡 |
| 复核 | 每日 HDBSCAN | 修正增量聚类的漂移 |

> 事件簇表 `events`：id, module, centroid_vector, member_article_ids[], source_count, first_seen, last_update, hotness。

## 6. 阶段五：AI 摘要与解读生成

### 6.1 产出结构（对应原型详情页字段）
```json
{
  "title": "钩子式标题（≤22字，可设问，不夸大）",
  "summary": ["3句话摘要：是什么/关键数据/最新进展"],
  "why_matters": "为何重要：对谁、有何影响（解读，1-2句）",
  "sources": [{"name":"", "level":"S/A/B", "url":""}],
  "facts": ["从原文抽取的关键事实/数字，逐条挂来源"],
  "disclaimer": "金融/宏观类自动附：非投资建议"
}
```

### 6.2 摘要 Prompt（防幻觉核心）
```
你是「热读」的资深财经科技编辑。基于【仅以下多源原文】生成结构化摘要，服务于投资人/创业者/产品经理。

严格规则：
1. 只使用原文出现的事实与数字，禁止编造或外推任何未提及的内容。
2. 数字必须与原文一致；原文没有的数字，相应字段留空。
3. 区分"事实"与"解读"：summary/facts 只写事实；why_matters 是你的解读但不得引入新事实。
4. 标题可用设问/悬念增强吸引力，但不得夸大或制造误导（不做标题党）。
5. 每条 facts 末尾标注来源序号 [1][2]。

【输出 JSON】严格按 schema：{title, summary[], why_matters, facts[], ...}

【多源原文】
[1]（{level}）{source_name}：{content_1}
[2]（{level}）{source_name}：{content_2}
...
```

### 6.3 模型策略（控成本）
| 用途 | 模型 | 理由 |
|---|---|---|
| 分类、去重判定 | 小模型 / 开源自部署 | 量大、简单，省成本 |
| 摘要/解读 | 中大模型（云 API） | 质量优先，是核心价值 |
| 深度解读（付费） | 大模型 + 更长上下文 | 付费内容，质量拉满 |

- **温度**：摘要类 temperature ≤ 0.3，降低发散。
- **结构化输出**：用 JSON mode / function calling 强约束 schema。
- **缓存**：同事件未更新不重复生成。

## 7. 阶段六：重要性评分与排序

### 7.1 评分公式（沿用 PRD §2.2 并细化）
```
importance = 0.4 × 来源等级权重(max)
           + 0.3 × min(多源交叉数 / 5, 1)
           + 0.2 × 归一化传播热度
           + 0.1 × 时效衰减
时效衰减 = exp(-Δt / τ)，τ≈12h
```
- **热度**来自 B 级社交信号 + App 内点击/收藏回流（冷启动先用外部热度）。
- **榜单**：Redis ZSet 按模块与全站各维护一份，分数即 `importance × 100`，实时刷新支撑首页/频道 TOP10。

### 7.2 排序产物
- 首页：全站 importance 降序 + 模块配额（避免某模块刷屏）。
- 频道：模块内 importance 降序。
- 热榜趋势（▲/新/—）：对比上一次快照的排名变化计算。

## 8. 阶段七：防幻觉校验与人工质检

### 8.1 自动护栏（发布前机器卡点）
| 校验 | 规则 | 不通过处理 |
|---|---|---|
| 引用校验 | 每条 facts 的数字能在原文中匹配到 | 命中失败 → 标红进人工 |
| 来源校验 | sources 非空且含可访问 url | 缺失 → 拦截 |
| 数字一致性 | 抽取摘要中的数字，比对原文 | 不一致 → 拦截 |
| 合规校验 | 敏感词 / 荐股 / 违规表述 | 命中 → 拦截 + 告警 |
| 金融免责 | 金融/宏观类强制附免责声明 | 自动补全 |

### 8.2 人工质检（CMS）
- **分级抽检**：S 级/付费/置顶内容 **必检**；普通内容按比例抽检。
- **后台能力**：审核通过/打回、编辑标题与摘要、调整模块/标签、置顶/下架、手动合并或拆分事件、触发推送。
- **反馈闭环**：人工修正样本回流，定期优化 Prompt 与规则（不一定微调模型，先迭代 Prompt）。

## 9. 调度、数据模型与可观测

### 9.1 调度（Celery 起步）
```
beat 定时   → 采集任务（按信源频率）
chain 编排  → 清洗 → 去重 → 分类 → 聚类 → 摘要 → 评分 → 入待审
独立 worker → 重活（LLM 调用）单独队列，限流 + 重试 + 退避
每日 cron   → HDBSCAN 复核聚类、早报组装、指标统计
```

### 9.2 核心数据表
| 表 | 关键字段 |
|---|---|
| `sources` | id, name, level(S/A/B), type, weight, enabled |
| `raw_articles` | id, source_id, url, title, content, raw_hash, embedding, published_at, fetched_at, status |
| `events` | id, module, tags[], centroid_vector, source_count, hotness, importance, status, first_seen, last_update |
| `event_articles` | event_id, article_id（多源归属） |
| `event_contents` | event_id, title, summary[], why_matters, facts[], deep_content(付费), version |
| `review_logs` | event_id, reviewer, action, before/after, time |

### 9.3 可观测指标（埋点）
- 各阶段处理量/失败率/时延；LLM 调用量与成本；去重率、串卡率；人工打回率；端到端时延。
- 失败可断点重跑（按 status 重入队）；LLM 调用全量留痕（输入/输出/版本）便于追责与回放。

## 10. 成本与风险

### 10.1 成本控制
- 分层用模型（小模型扛量、大模型保质）；规则兜底减少 LLM 调用；同事件缓存不重复生成；非高优内容批处理。

### 10.2 风险与应对
| 风险 | 应对 |
|---|---|
| LLM 幻觉/错数字 | 引用校验 + 数字比对 + 事实/解读分区 + 人工必检高优 |
| 串卡（不同事件聚成一个） | 保守阈值 + 每日 HDBSCAN 复核 + 人工可拆 |
| 信源版权 | 摘要+回源、不全文转载、法务确认授权策略 |
| 金融合规 | 免责声明、禁荐股敏感词、人工卡点 |
| 单点信源失效 | 多源冗余、信源健康监控、断流告警 |
| 成本失控 | 调用量与成本看板、限流、模型分层 |

## 11. M1 落地顺序（4 周建议）

| 周 | 交付 | 验收 |
|---|---|---|
| W1 | 采集 + 清洗 + 精确/语义去重 + 数据表 | 四模块种子源稳定入库，去重率 < 5% |
| W2 | 分类（规则+LLM）+ 增量聚类 | 分类准确率 ≥ 90%，同事件成单卡 |
| W3 | 摘要/解读生成 + 评分 + 榜单 | 摘要结构完整、不杜撰，榜单可用 |
| W4 | 防幻觉护栏 + CMS 质检 + 早报组装 | 端到端跑通，内部灰度日更，抽检合格 ≥ 90% |

> 出口 Gate：连续 3 天内部灰度，四模块日更内容抽检合格率 ≥ 90%，方可进入 M2 客户端联调。
