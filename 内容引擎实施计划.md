## 0. 现状 vs 目标差距

`pipeline_demo/run_pipeline.py` 当前是单进程、一次性、无状态脚本：`fetch_all → classify → dedup_exact → cluster → summarize → score`，最终落 `output.json`。生产化需补齐五个维度：

| 维度 | Demo 现状 | 生产目标（依据方案） |
|---|---|---|
| 调度 | 手动 `python3 run_pipeline.py` 跑一次 | Celery beat 定时 + chain 编排 + 重试退避 |
| 存储 | 内存 list + `output.json` | PostgreSQL + pgvector + Redis 榜单 |
| 算法 | 规则分类、Jaccard 去重/聚类 | LLM 分类、Embedding 语义去重(cos≥0.92)/聚类(cos≥0.86) |
| 质量护栏 | 无 | 引用校验 / 数字比对 / 合规 / 人工质检队列 |
| 可观测 | print 控制台 | **[已完成]** 阶段埋点（`pipeline_runs`）、统一日志、LLM 成本核算、报表看板（`/admin/metrics`） |

### 三条不可违背的铁律（贯穿全部阶段）
1. **不杜撰**：数字、引语、事实必须来自原文，缺失留空。
2. **可回溯**：每条事件挂得到原文链接与来源等级，事实与解读分区。
3. **可降级**：任一 AI 环节失败可回退到规则兜底 / 人工待审，绝不让脏内容直接发布。

## 2. 阶段 1：采集 + 去重生产化（对应方案 W1）

>
> 出口标准：四模块种子源稳定入库，去重率 < 5%；HTTP API 可被 iOS 客户端拉取每日简报、事件详情、信息流。

| # | 步骤 | 产出 / 验收 |
|---|---|---|
| 1.1 | **[已完成]** SimHash 精确去重：`utils.simhash64` + `clean.py` 时间窗汉明距离≤3 判重 | `raw_articles.simhash` 字段 + 5 个单测 |
| 1.2 | **[已完成]** 信源健康监控 + 断流告警：`source_health` 表 + collect 端写入与阈值告警 | 信源可用性可观测，3 个单测 |

## 7. 关键风险与建议

- **顺序铁律**：M1 内容引擎必须先于 M2 客户端联调就绪，否则 App 无内容可读。
- **起步顺序**：先做阶段 0 + 1（骨架 + DB + 采集去重），这是地基；阶段 2/3 算法可并行预研。
- **宏观短板**：当前 `output.json` 仅 4 条 macro，阶段 1.3 须优先补宏观信源。
- **成本控制**：模型分层（小模型扛量、大模型保质）、规则兜底减少 LLM 调用、同事件缓存。
